Design review — suivi-vues

Tableau de bord de veille des vues/écoutes · Streamlit · 8 juillet 2026 · destinataire : Claude Code (implémentation). Toutes les recommandations sont réalisables en Streamlit natif + CSS léger, sauf mention contraire.

0. Système transversal (à faire en premier)

  • Thème — dans .streamlit/config.toml : primaryColor="#33618F", backgroundColor="#FAFAF9", secondaryBackgroundColor="#F2F1EE", textColor="#1F2A37", font="sans serif". Blancs légèrement chauds, jamais de gris saturé.
  • Supprimer les emoji d'icônes (📊 ➕ 🔗 📥 🔄). Ils varient selon l'OS et donnent un ton non institutionnel. Nav en texte seul via st.Page(..., icon=None), ou icônes Material (icon=":material/monitoring:") si un repère visuel est voulu. Retirer l'emoji des titres de page (st.title sans emoji).
  • Copy — renommer : « Ajouter une vue » → Ajouter un relevé (on saisit un relevé daté, pas une vue) ; « Gérer les URLs » → Contenus & URLs ; « Importer un fichier » → Importer des relevés. Bannir les libellés anglais résiduels : page de connexion Login/Username/Password → Connexion / Identifiant / Mot de passe ; uploader → st.file_uploader("…") ne permet pas de traduire « Drag and drop », ajouter la consigne française en help/légende au-dessus et masquer le texte anglais est du CSS fragile — au minimum encadrer d'une consigne FR claire.
  • Couleur = contenu (choix validé) : une couleur stable par contenu, déclinée sur toutes ses URLs. Palette catégorielle sobre, même luminosité/chroma : #33618F, #8A6136, puis oklch(0.50 0.09 h) en variant la teinte (140, 320, 90…). L'appliquer à la fois aux courbes Plotly (color_discrete_map) et aux barres. La plateforme est indiquée par le texte, jamais par la couleur.
  • Nombres — format FR (espace fine comme séparateur de milliers), tabular-nums via CSS léger sur les tables/metrics.

1. Connexion (01)

  • Formulaire pleine largeur ~1300 px : le contraindre à ~420 px centré (st.columns([1,1,1]), formulaire dans la colonne du milieu) avec le nom du produit au-dessus.
  • Le message « Veuillez vous connecter » en st.info bleu ressemble à une erreur douce ; c'est un état normal. Le remplacer par un sous-titre statique sous le titre. Réserver les alertes aux vrais échecs (st.error("Identifiant ou mot de passe incorrect")).
  • Bouton Se connecter en type="primary", pleine largeur du formulaire ; soumission sur Entrée via st.form.

2. Tableau de bord (02, 07) — voir maquettes 1a / 1b / 1c

Diagnostic : la hiérarchie filtres → courbe → comparaison → audit existe mais tout a le même poids visuel ; la courbe démarre à ~40 % de la hauteur d'écran ; la légende Plotly à droite mange un tiers de la largeur ; aucun contexte (nb de contenus, dernière collecte).

  • Titre + ligne de contexte : « 2 contenus suivis · 4 URLs · dernière collecte le 19 janv. » (st.caption).
  • Filtres sur une seule rangée st.columns(3), libellés courts ; « Regrouper les courbes par » → Une courbe par.
  • Courbe : légende horizontale au-dessus du graphe (legend=dict(orientation="h", y=1.12)) pour rendre la largeur au tracé ; grille horizontale seule ; axe Y sans titre « Vues » redondant si le titre de section le dit ; marqueurs uniquement au dernier point.
  • « Dernières vues par plateforme » → Dernier relevé par plateforme, barres horizontales avec valeur affichée, colorées par contenu.
  • Table d'audit : en-têtes FR lisibles (pas platform_name, collection_method) via st.dataframe(column_config=…) ; source en toutes lettres (Collecte auto / Saisie manuelle / Import CSV).
  • Rôle lecteur (07) : la sidebar quasi vide est déroutante. Afficher le nom du produit + « Lecteur · consultation seule » en badge, ou passer la nav en haut (voir 1c) pour que lecteur et admin partagent la même structure.

3. Ajouter un relevé (03)

  • Usage fréquent → optimiser la vitesse : st.form avec soumission Entrée ; date par défaut = aujourd'hui ; après enregistrement, conserver l'URL sélectionnée et vider seulement la valeur (saisie en série).
  • « Nombre de vues » à 0 par défaut invite l'erreur : préremplir avec le dernier relevé connu de l'URL et l'afficher en aide (« Dernier relevé : 510 le 19 janv. ») — c'est le vrai garde-fou contre les régressions de compteur.
  • La case « Il s'agit d'un ajustement… » est ambiguë : la déduire automatiquement (si une valeur existe déjà pour cette date, basculer en mode ajustement et l'annoncer par un st.warning doux) plutôt que la demander.
  • Confirmation : st.toast("Relevé enregistré — Épisode 12, 19 janv., 512 vues").

4. Contenus & URLs (04)

  • L'ordre des onglets inverse le modèle mental : on crée d'abord un contenu, puis on y rattache des URLs. Mettre Créer un contenu en premier onglet, ou fusionner : dans le formulaire d'URL, l'option « + Nouveau contenu » ouvre la création inline.
  • Table « URLs suivies » : en-têtes FR, URL raccourcie (domaine + …), colonne « Méthode » en libellé (Auto / Manuel) avec pastille discrète, tri par contenu.
  • « Méthode de collecte » ne devrait pas être un choix libre : elle découle de la plateforme (YouTube/PeerTube → auto, sinon manuel). L'afficher en champ désactivé ou en simple mention.

5. Importer des relevés (05)

  • Écran trop nu : documenter le format attendu (colonnes, exemple) dans un st.expander("Format attendu") + lien de téléchargement d'un modèle CSV (st.download_button).
  • Après dépôt : aperçu des 5 premières lignes + rapport de validation (n lignes valides, n erreurs avec motif) avant écriture — étape Vérifier puis confirmer.

6. Collecte automatique (06)

  • Ajouter les colonnes utiles : Dernière collecte (date + valeur) et Statut (OK / échec) — c'est la question que l'admin vient poser à cette page.
  • Pendant la collecte : st.status("Collecte en cours…") puis résumé (« 2 URLs collectées, 0 échec ») ; horodater « Dernière exécution » près du bouton.
  • Le bouton violet est hors palette : type="primary" suffit (reprendra le bleu du thème).

7. Réponses aux questions de la review

  • Hiérarchie du tableau de bord — pas claire aujourd'hui (tout au même poids, courbe trop basse). Corrigée par les maquettes : contexte en tête, filtres compacts, courbe dominante, audit en dernier (replié en 1c).
  • Code couleur — la couleur par URL ne passe pas à l'échelle (11 plateformes × n contenus). Couleur par contenu + plateforme en texte : lisible et stable.
  • Formulaire de saisie — assez rapide une fois : soumission Entrée, valeurs préremplies, saisie en série (voir §3).
  • Distinction des rôles — insuffisante (sidebar vide côté lecteur). Badge de rôle explicite + structure de page identique entre rôles.
  • Accessibilité (WCAG AA) — points à corriger : gris des libellés ≥ 4.5:1 (utiliser #5D6B79 sur blanc, pas plus clair) ; ne jamais coder l'information par la couleur seule (étiquettes texte sur les courbes) ; cibles cliquables ≥ 44 px (boutons pleine hauteur) ; label explicite sur chaque widget ; contraste des courbes sur fond blanc ≥ 3:1 (les deux teintes proposées passent).

Hors périmètre Streamlit natif

Non recommandé sans développement significatif : nav horizontale custom stylée (1c — possible via st.tabs mais rendu approximatif ; la sidebar de 1a/1b est le choix sûr), traduction du texte interne de st.file_uploader, tables avec pastilles colorées riches (préférer column_config simple).
